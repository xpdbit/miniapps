# SuperTask 远景规划

> 本文档记录 SuperTask 的长期演进设想，涵盖定位、阶段性演进、核心功能方向、以及与现有工具的边界。

---

## 一、核心定位

### SuperTask 是什么

**代码认知层** — 不替代开发者的编码能力，而是放大开发者对代码库的感知力。

### SuperTask 不是什么

| 角色 | 说明 |
|------|------|
| ❌ 不是执行加速器 | 它的重点是「发现该做什么」，而非「做得更快」 |
| ❌ 不是 PM 工具 | 任务跟踪和资源分配是 Jira/Linear 的职责 |
| ❌ 不是 CI 系统 | 自动化测试和部署是 CI/CD 的职责 |
| ❌ 不是代码审查替代品 | 最终决策权永远在开发者手中 |

---

## 二、解决的问题

### 2.1 单人维护的问题

个人项目（如本 monorepo）在维护期面临两个核心瓶颈：

**瓶颈 A：需求发现的认知税**

团队有 PM/QA 兜底需求发现，单人项目你同时是产品经理、开发、测试。面对 750+ 源文件、3 个项目各带独立后端，不可能每天手动扫一遍找问题。AI 主动探索承担了那部分「本应由多人团队分担的认知负载」。

**瓶颈 B：Agent 的持久化困境**

opencode 等 CLI agent 是 stateless worker——你给 prompt，它干活，干完退出。它没有「跑完一个任务后自己想想下一步该干啥」的能力。SuperTask 的本质是在 agent 外包一层 stateful loop，赋予它长期记忆和自我驱动的能力。

### 2.2 团队扩张后的新问题

| 问题 | SuperTask 的定位 |
|------|------------------|
| 核心成员难以掌握所有项目全貌 | 作为个人的代码感知层，定期输出项目摘要 |
| 新人上手要读一周代码 | 探索模式产出架构摘要、关键模块清单、已知问题 |
| 跨项目的共性问题没人发现 | AI 扫完 A 项目发现 B 项目有同样的 anti-pattern |
| 碎片化想法散落在各处 | 统一的 backlog inbox，每天过滤一次 |

## 三、演化路线

```
现在（单人维护）    →   小团队（3-5人）       →   公司（10+人）
──────────────────────────────────────────────────────────────────
SuperTask + opencode     超级个体的代码嗅觉          代码健康巡检层
TUI 直用                    PM 工具双向同步            PM 工具双向同步
                            个人仍用它做主力          仅核心人员使用
                            探索提议辅助规划           定期技术债报告
```

### 阶段 1：单人模式（当前）

- **使用组合**：有明确想法时 opencode TUI 直接干；不确定做什么时 SuperTask 探索 + 审批 + 执行
- **核心能力**：Cycle 模式（探索→提议→审批→执行）
- **技术基础**：session checkpoint、worktree 隔离、agent 状态追踪、崩溃恢复

### 阶段 2：个体+PM 工具协同

- **新增能力**：PM 工具双向同步（webhook 接收工单、回写状态）
- **新增能力**：Agent agnostic 任务分派（不锁定 opencode）
- **不变**：个人仍用 Cycle 模式做主动探索

### 阶段 3：代码健康巡检层

- **新增能力**：可量化的代码健康仪表盘
- **新增能力**：自动化 pre-review gate
- **使用者**：核心成员/技术负责人，不要求开发团队全员使用
- **输出**：定期技术债报告，与 PM 工具对接

## 四、关键演进方向

### 4.1 Agent 无关的任务分派层

**现状**：opencode_runner.py 写死 opencode。

**目标**：抽象为「任务下发接口」，下游可接入任意 CLI agent。

```
SuperTask 编排层
      │
      ├── opencode    （当前支持）
      ├── Claude Code （未来）
      ├── Codex       （未来）
      └── Aider       （未来）
```

SuperTask 只负责任务管理（分派、追踪、状态同步）和准入质检（pre-review），不绑定具体 Agent 实现。开发者用自己顺手的 Agent 完成任务。

**架构变更**：
- 定义抽象的 `AgentRunner` 接口
- opencode 实现为其中一个 adapter
- 配置化选择默认 agent 类型

### 4.2 PM 工具双向同步

**现状**：无 PM 工具集成。

**目标**：SuperTask 与 PM 工具的分工清晰化。

```
PM 工具（任务跟踪）          SuperTask（代码认知层）
────────────────────         ────────────────────────
工单创建 / 排期              已发现的问题和机会
资源分配                     探索模式产出 backlog 素材
完成状态跟踪                  代码健康仪表盘
协作沟通                     pre-review 结果
```

**集成方式**：
- 接收：Jira/Linear webhook → SuperTask 自动转化为 `approved_queue` 任务
- 回写：任务完成状态 → PM 工具 API 更新对应工单
- 同步频率：事件驱动（非 polling），降低开销

### 4.3 代码健康量化仪表盘

**现状**：探索模式输出自然语言的任务提议。

**目标**：在提议的基础上，输出可量化的代码健康信号。

| 指标 | 说明 | 追踪方式 |
|------|------|----------|
| `as any` 密度 | 每千行中的类型逃逸数量 | AST 扫描趋势图 |
| 空 catch 覆盖率 | 无处理的异常捕获数量 | AST 扫描 |
| TODO 堆积率 | TODO/FIXME/HACK 注释的增减速 | git blame + grep |
| 过时依赖里程 | 超过 N 个版本未更新的依赖 | package.json vs latest |
| 测试覆盖率缺口 | 新增代码未被测试覆盖的部分 | 覆盖率 diff |

量化数据比自然语言建议更有说服力，也更容易被集成到团队工作流中。

### 4.4 自动化 pre-review Gate

**现状**：执行完成后依赖人工审查。

**目标**：开发者提交变更后，SuperTask 自动执行一轮 pre-review。

**pre-review 检查项**：
- 类型一致性（与项目 tsconfig 标准对齐）
- Anti-pattern 扫描（新增的 as any、空 catch）
- 项目规范一致性（命名、目录结构、导入风格）
- 跨项目影响分析（变更是否影响到共享模块）

pre-review 不阻塞合入，但生成报告附在任务记录中。开发者自行决定是否处理报告中的问题。

### 4.5 Cycle 模式深度化

方向不是「更快」，而是**「更准」**：

| 维度 | 当前 | 目标 |
|------|------|------|
| 探索质量 | 关键词匹配分类 | 基于项目历史的上下文感知 |
| 跨项目关联 | 独立扫描各项目 | 发现跨项目的共性问题 |
| 上下文记忆 | 每次探索从头开始 | 记住上一轮结论，避免重复提议 |
| 优先级判断 | 固定规则 | 基于代码变更频率和影响面 |

## 五、边界与原则

### 5.1 不做的事

- ❌ 不做 PM 工具 —— 不替 Jira/Linear 管理排期和资源
- ❌ 不做 CI 系统 —— 不替 GitHub Actions/Jenkins 跑测试和部署
- ❌ 不做代码审查替代品 —— AI 提议和 pre-review 是辅助，最终决策权在开发者
- ❌ 不做 IDE —— 不替 VSCode/Cursor 做代码编辑
- ❌ 不做全自动开发 —— 永远保留人的审批环节

### 5.2 设计原则

1. **提议与执行分离** — AI 可以提议，但决策必须经过人
2. **人类在环** — 每个关键节点都有人的介入点
3. **渐进采用** — 个人可用，团队可选，不强推
4. **工具无关** — 不锁定 Agent、不锁定 PM 工具、不锁定平台
5. **量化优先** — 能用数字说话不用自然语言

## 六、与竞品的关系

| 工具 | 定位 | 与 SuperTask 的关系 |
|------|------|----------------------|
| Batty | 终端原生并行 agent 执行 | 执行层互补，SuperTask 可接入 Batty 作为 runner |
| Superset | 桌面多 agent 编排工作区 | 平行竞争，SuperTask 的差异化是 Cycle 模式 |
| Emdash | 开源 worktree 多 agent 执行 | 同上，SuperTask 差异化在认知减负而非执行效率 |
| Capy | 云端多 agent 协作平台 | 场景不同，SuperTask 定位本地优先 |
| CrewAI/LangGraph | Agent 编排框架 | 不同层，SuperTask 可与之组合使用 |

SuperTask 的独特价值在于：不是「如何让 Agent 跑得更快」，而是**「如何让开发者知道该做什么」**。

---

## 七、与 opencode 的关系

当前 SuperTask 锁定 opencode 作为唯一的 agent runner。这是实用主义选择——本项目的所有工作流都基于 opencode，锁定它降低了实现复杂度。

未来方向是 Agent agnostic，但 opencode 始终是一等公民。
