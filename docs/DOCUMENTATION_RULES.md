# 文档编写规则

> 本文档定义 `.miniapps` 项目中所有文档的编写规范。
> 所有在 `docs/` 目录下新增或修改文档时，必须遵守此规则。

---

## 1. 文档分类与定位

### 1.1 三种文档类型

| 类型 | 位置 | 用途 | 谁写 |
|------|------|------|------|
| **项目文档** | `docs/` | 面向开发者（人类），描述项目结构/架构/使用方式 | 开发者 / AI |
| **Agent 知识库** | `AGENTS.md`（各目录） | 面向 AI Agent，描述代码约定/反模式/代码地图 | AI Agent |
| **内联文档** | 代码注释 (JSDoc / 行内) | 面向开发者（人类），解释具体函数/类的行为 | 开发者 / AI |

### 1.2 内容边界

| 内容 | 放哪里 |
|------|--------|
| 项目概述、技术栈、如何启动 | `docs/<project>/README.md` |
| 架构设计、数据流、子系统关系 | `docs/ARCHITECTURE.md`（跨项目）或 `docs/<project>/README.md`（项目内） |
| 开发指南、环境搭建、工作流 | `docs/<project>/DEVELOPMENT.md` |
| API 接口定义（稳定版） | `docs/<project>/API.md` |
| 数据库 Schema 说明 | `docs/<project>/DATABASE.md` |
| 部署流程、容器编排、域名配置 | `docs/deploy/README.md` |
| 设计方案、重构计划 | `docs/superpowers/specs/` |
| 代码约定、反模式列表、文件地图 | `AGENTS.md`（项目根或子目录） |
| 函数/类的用途、参数、返回值 | JSDoc 代码注释 |
| 函数内部复杂逻辑的解释 | 行内代码注释 |

### 1.3 边界红线

- ❌ `AGENTS.md` 中不要写如何启动项目（放 `docs/`）
- ❌ `README.md` 中不要写代码反模式（放 `AGENTS.md`）
- ❌ `docs/` 中不要写 AI Agent 才知道的内部约定
- ✅ 允许两者重复少量内容（如项目技术栈），但避免大段复制

## 2. 文件命名规范

### 2.1 命名规则

```
PascalCase.md      — 标准文档 (README.md, ARCHITECTURE.md)
UPPER_CASE.md      — 专有领域 (API.md, DATABASE.md, CHANGELOG.md)
kebab-case.md      — 规格/计划文档 (2026-05-10-ai-tavern-design.md)
```

### 2.2 文件清单

| 文件名 | 用途 |
|--------|------|
| `README.md` | **必须** — 项目概述、技术栈、核心功能、快速开始 |
| `DEVELOPMENT.md` | 可选 — 开发环境搭建、工作流、规范 |
| `API.md` | 可选 — 接口参考文档 |
| `DATABASE.md` | 可选 — 数据库设计说明 |
| `ARCHITECTURE.md` | **仅根目录** — 系统级架构 |
| `CHANGELOG.md` | **仅根目录** — 项目级变更日志 |
| `DOCUMENTATION_RULES.md` | **仅根目录** — 文档编写规则（当前文件） |
| `YYYY-MM-DD-<description>.md` | 设计规格/计划（放 `superpowers/specs/` 或 `superpowers/plans/`） |

### 2.3 每个项目的最小文档集

```
docs/<project>/
├── README.md       # 必须 — 项目概述 (≤80行)
├── API.md          # 建议 — API 参考
├── DATABASE.md     # 建议 — 数据库设计
└── DEVELOPMENT.md  # 建议 — 开发指南
```

- **Server 项目**：必须包含 README.md + API.md（路由表）
- **MiniApp 项目**：必须包含 README.md
- **Game1 MiniApp**：README.md 作为重构方案已超长，考虑拆分为 `DESIGN.md`

## 4. 写作风格规则

### 4.1 语言

- **所有文档使用简体中文**
- 专有名词（框架名、库名、术语）保留英文原文：`TypeScript`, `Prisma ORM`, `Zustand`
- 代码示例中的注释使用中文
- 避免中英文混写，用 "数据库" 而非 "数据 base"

### 4.2 格式

- **标题**：Atx 风格 (`##`)，句首大写，不用尾句号
- **表格**：必须包含表头分隔行 (`|---|---|`)
- **代码块**：必须标注语言 (` ```typescript `)
- **ASCII 图**：使用 ` ``` ` 包裹，保持对齐
- **强调**：使用 `**加粗**` 表示关键概念，` ` 表示代码/路径/命令

### 4.3 语气

- **客观、精确、简洁** — 不说废话
- **禁止**：营销语气、过度解释、情绪化表达
- **用** "必须/禁止/建议/可选" 而非 "最好/不要/应该"
- **不写** "这是一个非常强大的功能" — 让事实说话

### 4.4 链接规范

- 文档内部链接使用相对路径：`./API.md`
- 跨项目链接使用 `../`：参考根 AGENTS.md 时用 `../../AGENTS.md`
- 外部链接用完整 URL：`[Taro 文档](https://docs.taro.zone/)`

### 4.5 废弃文档处理

```
> 🚫 **废弃文档** — 本文档描述的是已弃用的 [旧架构名称] (2025)。
> 当前架构已完全迁移至 [新架构名称]。
> 如需参考旧内容，请查看 Git 历史中的此文件。
```

- 废弃文档**保留在仓库中**（不要删除），仅添加废弃标记
- 废弃标记放在文档第二行（标题下方）
- 在文档索引中标注「废弃」

## 5. 维护规则

### 5.1 更新规范

- 每次修改文档时，在文件末尾更新 `---` 后的修改记录：

```markdown
---

> 最后更新: 2026-05-12
> 修改: 新增 API 路由说明
```

- 重大变更（架构重构、API 重写）必须同时更新 `docs/ARCHITECTURE.md`
- 新增子项目时，必须同步更新：
  1. `docs/README.md`（文档索引）
  2. 创建对应项目的 `docs/<project>/README.md`
  3. 更新 `docs/ARCHITECTURE.md`（如涉及架构变更）

### 5.2 废弃流程

1. 在旧文档头部添加 `> 🚫` 废弃标记
2. 在 `docs/README.md` 的项目文档表中标注废弃
3. 创建新文档替代
4. 旧文档保留（不删除），供 Git 历史参考

### 5.3 质量门禁

每篇文档发布前检查：

- [ ] 标题清晰反映内容
- [ ] 没有过期信息（检查日期敏感的配置、版本号）
- [ ] 代码块标记了正确语言
- [ ] 表格有表头分隔行
- [ ] 链接可用（相对路径拼写正确）
- [ ] ≤ 文档类型对应的最大行数
- [ ] 没有「TODO」「待补充」等占位符（可用 `plan/` 跟踪）
- [ ] 没有与 `AGENTS.md` 重复的大段内容

## 6. 黄金模式（应保留的好习惯）

| 模式 | 示例 | 说明 |
|------|------|------|
| **废弃标记** | `> 🚫 **废弃文档**` | 清晰标记 + 替代方案说明，不删旧文档 |
| **表格化呈现** | 技术栈/路由表/服务层 | 结构化信息用表格，一目了然 |
| **ASCII 架构图** | ARCHITECTURE.md | 字符画图表，版本控制友好 |
| **依赖链可视化** | `T1→T2→T3→...` | 任务/重构依赖关系用箭头表示 |
| **验证清单** | plans 末尾的 Checklist | 可执行的验证步骤 |
| **章节编号** | `一、/1.1/1.2` | 复杂文档按章节编号，层次清晰 |
| **命令即用** | 代码块中的命令可直接复制执行 | 代码块包含完整执行路径 |
| **相对路径引用** | `./API.md` 而非绝对路径 | 所有路径引用使用相对路径 |

## 7. 常见反模式

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| **README 过长** | 单个 README > 200 行 | 拆分为多个专题文档 |
| **空文档** | 仅标题无内容 | 标记为「待补充」或删除 |
| **无日期信息** | 文档无版本/更新日期 | 末尾添加修改记录 |
| **代码与文档脱节** | API 文档与实际路由不匹配 | 保持同步更新 |
| **废弃文档无标记** | 旧架构文档无废弃提示 | 添加 `> 🚫` 标记 |
| **重复内容** | 多个文档写相同信息 | 维护单一事实来源，其他引用 |
| **文档缺失** | 子项目无对应文档 | 创建 README.md (≥30行) |
| **重复内容** | 同一文件内重复段落（如 deploy/README.md 架构图出现两次） | 维护唯一版本，删除重复 |
| **空文档/残档** | 仅有标题无实际内容（如 Items.md） | 补充内容或标记「待补充」 |
| **平台命令不匹配** | docs 中使用 bash 命令但环境是 Windows | 标注命令执行环境；跨平台命令用标准 shell |
| **绝对路径** | 文档中出现 `E:\.Code\.miniapps\...` 等绝对路径 | 全部转换为相对路径 |
| **表格内容不对齐** | 同一类型表格在不同文档中列不同 | 同类文档使用相同的表结构 |
