# 文档编写规则

> **状态**: current（内容已整合至 `docs/standards/` 目录）
> **更新**: 2026-05-26

> 所有在 `docs/` 目录下新增或修改文档时，必须遵守此规则。

## 一、文件命名规范

### 1.1 命名规则

| 格式 | 用途 | 示例 |
|------|------|------|
| `README.md` | 项目概述、技术栈、快速开始 | `apps/ftg/client/README.md` |
| `API.md` | API 接口参考 | `apps/ftg/server/API.md` |
| `VISION.md` | 长期演进规划 | `tools/supertask/VISION.md` |
| `kebab-case.md` | 规格/计划文档 | `2026-05-10-ai-tavern-design.md` |

### 1.2 保留文件名

| 文件 | 位置 | 用途 |
|------|------|------|
| `README.md` | 每个项目目录 | **必须** — 项目概述 |
| `API.md` | server 目录 | **建议** — API 参考 |
| `rules.md` | `docs/` 根目录 | **唯一** — 编写规则 |
| `edge.md` | `docs/` 根目录 | **唯一** — 文档边界 |
| `urls.md` | `docs/` 根目录 | **唯一** — URL 清单 |
| `ARCHITECTURE.md` | `docs/` 根目录 | **唯一** — 系统架构 |

## 二、文档格式规范

### 2.1 标题层级

- 使用 Atx 风格：`#` → `##` → `###` → `####`
- `#` 仅用于文档标题（每文档一个）
- 句首大写，不用尾句号

### 2.2 代码块

- 必须标注语言：```typescript
- 命令代码块可直接复制执行
- ASCII 架构图用 ``` 包裹，保持对齐

### 2.3 表格

- 必须包含表头分隔行：`|---|---|`
- 同类文档使用相同表结构
- 长内容使用 `---` 分隔线分段

### 2.4 链接

- 内部链接使用相对路径：`./API.md`
- 跨项目使用 `../` 上溯
- 外部链接使用完整 URL

### 2.5 强调

- `**加粗**` — 关键概念
- `` 代码反引号 — 路径、命令、变量名
- `>` 引用 — 废弃标记、注意事项

## 三、写作风格

### 3.1 语言

- 所有文档使用简体中文
- 专有名词保留英文原文：`TypeScript`, `Prisma ORM`, `Zustand`
- 代码示例中的注释使用中文
- 语气客观、精确、简洁

### 3.2 用词规范

| 规范 | 正确 | 错误 |
|------|------|------|
| 强制性 | 必须 / 禁止 | 最好 / 不要 |
| 建议性 | 建议 / 可选 | 应该 |
| 客观描述 | 提供 / 返回 | 这是一个非常强大的功能 |

### 3.3 版本标注

- 日期敏感的配置、版本号必须在文档末尾标注更新日期
- 已废弃的架构使用标记：`> 🚫 **废弃文档**`

## 四、项目命名约定

| 概念 | 约定 | 示例 |
|------|------|------|
| 项目名 | 英文简称 | FTG, Game1, Tavern |
| 端口 | 数字 + 说明 | 3000 (FTG), 3001 (Game1/Admin), 3002 (Tavern) |
| 环境变量前缀 | 大写 + 下划线 | `TARO_APP_`（前端）, 自定义（server） |
| 文件路径 | 相对于仓库根 | `apps/ftg/client/` |
| Prisma 版本 | v+主版本 | v6.19 |
| 语言 | Taro + React | Taro 4.x + React 18 |

## 五、文档分类与边界

| 内容 | 存放位置 |
|------|----------|
| 项目概述、技术栈、快速开始 | `docs/<project>/README.md` |
| 架构设计、子系统关系 | `docs/ARCHITECTURE.md` |
| API 接口定义 | `docs/<server>/API.md` |
| 部署流程、容器编排 | `docs/ops/deploy.md` |
| 设计方案 | `plan/specs/` |
| Agent 工作计划 | `.sisyphus/plans/` |
| 代码约定、反模式列表 | 各目录 `AGENTS.md` |
| 函数/类的行为 | JSDoc 代码注释 |

## 六、维护规则

### 6.1 更新记录

每次修改文档时，在文件末尾添加修改记录：

```markdown
---

> 最后更新: 2026-05-18
> 修改: 新增 XXX 说明
```

### 6.2 废弃流程

1. 在旧文档头部添加 `> 🚫` 废弃标记
2. 保留旧文件不删除（供 Git 历史参考）
3. 创建新文档替代

### 6.3 质量门禁

发布前检查：

- [ ] 标题清晰反映内容
- [ ] 没有过期信息
- [ ] 代码块标注了正确语言
- [ ] 表格有表头分隔行
- [ ] 链接可用（相对路径拼写正确）
- [ ] 没有「TODO」「待补充」占位符
- [ ] 没有与 `AGENTS.md` 重复的大段内容
