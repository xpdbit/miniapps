# SuperTask — AI 自主开发监督系统

基于 Python GUI 程序，循环驱动 AI 代理（opencode）自动探索项目、提出任务，经人工审批后执行开发、文档、推送等操作。

## 定位

SuperTask 不是执行加速器，而是**认知减负器**。

个人项目的瓶颈往往不是写代码的速度，而是**不知道改什么**和**AI 难以持久化运行**两个问题：

| 问题 | 影响 | SuperTask 的解法 |
|------|------|------------------|
| 不知道改什么 | 面对 750+ 源文件，很难主动发现技术债和改进点 | **探索模式** — AI 定期扫描代码库，生成任务提议清单 |
| AI 难以持久化 | opencode 每次会话结束就清空上下文，无法持续追踪 | **状态化循环** — 持久化 YAML 状态 + session checkpoint，崩溃可恢复 |

SuperTask 是面向"半自动维护"场景的工具：AI 主动发现，你审批决策，AI 落地执行。它适合维护期的多项目管理、低决策成本的碎片化改进、以及开发者疲劳期的低认知启动。

## 快速开始

```bash
pip install -r requirements.txt
python gui/main.py
# 或双击 miniapps_supertask.bat
```

## 新特性: 二次评估

在「提议与工作」面板点击 **二次评估** 按钮，系统会用高级模型（默认 `deepseek-v4-pro`）重新评估所有待审批提议：

- **真实性验证** — 确认每项提议描述的问题在代码库中真实存在，剔除假阳性
- **同类合并** — 将"空 catch 补日志"等重复提议合并为一项
- **优先级重评** — 让高级模型重新判断严重程度（flash 的 P2 可能实际是 P3）
- **难度预估** — 为每条提议标注修复难度（quick/normal/heavy）

评估模型可在「配置 → 通用设置 → 评估模型」中自定义。

## 工作流

1. **探索与提议** — AI 遍历项目，生成任务提议列表
2. **人工审批** — 在 GUI 中选中任务行，批准后进入执行队列
3. **逐项执行** — AI 按队列顺序执行，状态实时更新
4. **收尾** — 自动更新 AGENTS.md、docs/，git push

## 工作模式

SuperTask 支持两种工作模式，覆盖从"不知道做什么"到"明确知道要做什么"的完整 spectrum：

### Cycle 模式（主动探索）

```
[探索] AI 扫描项目 → [提议] 生成任务列表 → [审批] 人工勾选 → [执行] AI 逐项开发 → [循环]
```

适用于：不确定今天该做什么、项目太久没巡检、想发现技术债。

### Dispatch 模式（按需执行）

```
[收到任务] → [执行] AI 立即开工 → [审查] 人工验收 → [完成]
```

适用于：有明确任务、PM 派工、紧急修复。任务来源可以是手动输入，也可以来自 PM 工具的 webhook 同步。

这两种模式可无缝切换 —— Cycle 模式审批的任务自动进入 Dispatch 模式的执行队列。

## 任务规划

任务规划 Tab（导航栏「任务规划」）提供 AI 驱动的任务细化功能：

- **AI 规划** — 用户输入高级任务描述，AI 生成 3-5 个关键规划问题
- **多选问答** — 每个问题以卡片形式展示 2-4 个选项（支持单选/多选）
- **规约生成** — 用户选择答案后，自动组合生成精细化的任务规约

## 控制面板

控制面板采用上下 35:65 分割布局：

**上侧 35%** — 左右 50:50 分割
- **左侧**: 统计数字（待审批/排队中/已完成/已失败）+ Agent 状态条（阶段/模型/耗时，耗时每秒实时刷新）
- **右侧**: 项目选择（可编辑下拉框）+ 5 个 4:3 方块卡片按钮水平排列（持续探索/执行队列/更新待审批/检查成果/更新文档并推送）+ Agent 控制（暂停/继续）

**下侧 65%** — 左右 50:50 分割
- **左侧**: 彩色 HTML 运行日志
- **右侧**: 工作队列检视表格

## Agent 状态面板

Agent 状态监控页面（导航栏「Agent 状态」）和控制面板顶部均实时显示 agent 运行状态：

- **阶段状态条** — 当前阶段名称 + 模型名称 + 实时耗时（每秒自增刷新）
- **Agent 类型筛选** — 通过下拉框按主 Agent 类型（如 sisyphus/sisyphus-pro）和子 Agent 类型（explore/oracle 等）筛选显示
- **Agent 详情表格** — 6 列信息：
  - Agent / Sub-agent — 主 agent 青色，子 agent 灰色 + `└─ ` 层级前缀
  - 类型 — 如 explore、oracle、librarian
  - 模型 — 当前使用的 AI 模型
  - 状态 — 带图标（● 运行中 / ⏸ 暂停 / ✓ 完成 / ✗ 错误）
  - Preview — 任务描述预览（截断 40 字）
  - 耗时 — 每秒实时刷新
- **视觉分组** — 不同类型的 agent 组之间以分隔行隔开

## 控制面板按钮

手动操作区采用 5 个 4:3 方块卡片水平排列，每项操作有独立边框 + hover 高亮效果：
- 持续探索、执行队列、更新任务、检查成果、检查修复、部署云端、**更新文档并推送**（Github）

AI 提议面板额外提供 **二次评估** — 用高级模型审核提议列表。

## 日志命名

| 文件 | 内容 |
|------|------|
| `{YYYYMMDD}.md` | 系统运行日志（info/error/decision/approved） |
| `{YYYYMMDD}_terminal.md` | agent 每次 prompt 调用的完整输入输出 |

## 约束

- 永不自动停止（除非手动暂停）
- 提议与执行分离
- 异常不中断循环
- 连续失败 2 次自动阻塞

## 远景

> 以下是对 SuperTask 长期演进的设想，记录了当前设计和未来方向的思考。

### 核心定位

SuperTask 的终极定位是**代码认知层**——不替代开发者的编码能力，而是放大开发者对代码库的感知力。

在单人模式下，它解决"不知道改什么"的问题；
在团队模式下，它解决"核心成员难以掌握所有项目全貌"的问题。

### 演化路线

```
现在（单人维护）    →   小团队（3-5人）       →   公司（10+人）
──────────────────────────────────────────────────────────────
SuperTask + opencode     超级个体的代码嗅觉          代码健康巡检层
                           PM 工具双向同步            PM 工具双向同步
                           个人仍用它做主力          仅核心人员使用
                                                    产出技术债报告
```

### 关键演进方向

**1. Agent 无关的任务分派层**
现状锁定 opencode，未来应抽象为"任务下发接口"，下游可接入 opencode、Claude Code、Codex、Aider 等任意 CLI agent。SuperTask 只负责任务管理和准入质检，不绑定具体 Agent 实现。

**2. PM 工具双向同步**
通过 webhook 接收 Jira/Linear 的工单，自动转化为 `approved_queue` 中的任务；任务完成后回写状态到 PM 工具。SuperTask 与 PM 工具的分工：SuperTask 感知代码状态、发现机会；PM 工具跟踪已确认的任务进度、分配资源。

**3. 代码健康量化仪表盘**
不只要"发现可改进的问题"，更要输出可量化的代码健康信号：`as any` 数量趋势、类型逃逸密度、过时依赖清单、TODO 堆积率。量化数据比自然语言建议更有说服力。

**4. 自动化 pre-review Gate**
SuperTask 可在开发者提交变更后，自动执行代码规范检查、类型一致性校验、anti-pattern 扫描，将低级问题拦截在人力审查之前。审查者只关注逻辑正确性和设计合理性。

**5. Cycle 模式深度化**
方向不是"更快"，而是"更准"：探索产出质量、跨项目关联发现能力、上下文记忆（记住上一轮探索的结论，避免重复提议）。

### 边界红线

- ❌ SuperTask 不替代 PM 工具 —— 任务跟踪和资源分配是 Jira/Linear 的职责
- ❌ SuperTask 不替代 CI —— 自动化测试和部署是 CI/CD 的职责
- ❌ SuperTask 不替代人审 —— AI 提议和 pre-review 是辅助，最终决策权在开发者
- ✅ SuperTask 专精于：代码状态感知 + 任务发现 + 执行编排 + 质量准入
